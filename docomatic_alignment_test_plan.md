# План разработки теста соответствия документации UDE и Docomatic (Mock SDK)

Этот план описывает этапы создания автоматизированного теста, верифицирующего 100% семантическую и структурную идентичность результатов генерации Universal Documentation Engine (UDE) историческому выводу наследуемой системы Docomatic на примере Mock SDK (C++, C#, Java).

---

## 1. Подготовка папок превосходства (Allowances)

Для фиксации намеренных расхождений (превосходство UDE, измененная структура сайдбара и т.д.) создается трехуровневая изолированная структура папок в репозитории `engine`:

1. Создать базовую директорию: `engine/tests/docomatic_allowances/`
2. Создать вложенные папки по маске `<ParserID>_<NormalizerID>_<RendererID>`, точно отражающие конфигурации задействованных пайплайнов:
   * `doxygen_cpp_comment_html_cpp/`
   * `doxygen_csharp_comment_html_csharp/`
   * `doxygen_java_comment_html_java/`
   * `doxygen_cpp_comment_hugo_cpp/` (для тестов сравнения с Hugo Markdown)
3. Создать в каждой папке начальные JSON-файлы допущений для Mock SDK:
   * `mock_sdk_cpp.json` (в папках для C++)
   * `mock_sdk_cs.json` (в папках для C#)
   * `mock_sdk_java.json` (в папках для Java)

---

## 2. Настройка автоматического игнорирования Git

Чтобы файлы допущений для реальных коммерческих SDK случайно не попали во внешний репозиторий, настраивается строгий фильтр в файле `.gitignore` субмодуля `engine`:

1. Открыть `engine/.gitignore`.
2. Добавить следующие строки в секцию тестовых ассетов:
   ```gitignore
   # Игнорировать любые локальные файлы превосходства/допущений по умолчанию
   tests/docomatic_allowances/**/*

   # Исключить из игнорирования и разрешить коммитить только файлы для Mock SDK
   !tests/docomatic_allowances/**/mock_sdk_*.json
   ```

---

## 3. Разработка тестового скрипта `tests/test_docomatic_alignment.py`

Тест реализуется на базе фреймворка `pytest`. Чтобы изолировать логику сравнения от особенностей парсинга различных форматов вывода UDE (HTML, Hugo Markdown, а в будущем — VitePress, Docusaurus и т.д.), вводится архитектура **загрузчиков контента (Output Loaders)**.

### Шаг 3.1. Интерфейс загрузчиков результатов UDE (`BaseUdeOutputLoader`)
Разработать абстрактный базовый класс `BaseUdeOutputLoader`, определяющий контракт для чтения результатов генерации UDE:
* `load_sidebar(self, output_dir: Path) -> dict`: Возвращает иерархическое дерево сайдбара во внутреннем стандартизованном формате тестов.
* `load_entities(self, output_dir: Path) -> dict[str, Path]`: Возвращает сопоставление ID сущностей и путей к их файлам.
* `load_entity_content(self, file_path: Path) -> dict[str, Any]`: Считывает контент страницы и возвращает разобранные семантические блоки (Brief, Description, таблицу параметров с именами/типами/описаниями).

### Шаг 3.2. Реализация специализированных загрузчиков UDE
1. **`HtmlOutputLoader`** (для формата `html`):
   * Считывает файл `nav_data.js` из папки вывода, парсит JSON-объект меню для извлечения сайдбара.
   * Обходит `.html` файлы сущностей, используя `lxml`/`BeautifulSoup` для извлечения текстовых блоков из HTML-разметки (селекторы заголовков, абзацев `<p>` и таблиц параметров).
2. **`HugoMarkdownOutputLoader`** (для формата `hugo_markdown`):
   * Рекурсивно обходит `.md` файлы в папке вывода.
   * Извлекает из YAML-фронтматтера метаданные сайдбара (`sidebar_position`, `weight`, `parent`) для реконструкции дерева меню.
   * Парсит тело Markdown-файлов для извлечения семантических блоков контента.

### Шаг 3.3. Модуль загрузки допущений (Allowances Engine)
Разработать класс `AlignmentAllowances`, который:
* Считывает конфигурацию запущенного теста.
* Вычисляет путь к файлу допущений: `tests/docomatic_allowances/{ParserID}_{NormalizerID}_{RendererID}/mock_sdk_{lang}.json`.
* Предоставляет методы для валидации найденных нестыковок: `is_allowed_mismatch(type, path, key, value)`.

### Шаг 3.4. Тестирование структуры сайдбара (Sidebar Tree Match)
* **Docomatic**: Считывает эталонный файл `contents.html` из архива вывода Docomatic. Парсит элементы с классами `.ContentFolder` (папки разделов), `.ContentGroup` (группы сущностей) и `.ContentTopic` (отдельные страницы) для построения иерархического дерева.
* **UDE**: Вызывает `loader.load_sidebar(output_dir)` соответствующего загрузчика UDE.
* **Сравнение**: Сверяет вложенность, порядок следования и названия пунктов меню. Любые отклонения должны либо отсутствовать, либо содержаться в секции `sidebar_allowances` загруженного JSON.

### Шаг 3.5. Проверка полноты сущностей (Completeness check)
* Выполняется симметричное сравнение списков сущностей, полученных от Docomatic и загрузчика UDE (`loader.load_entities`).
* Обнаруженные расхождения (например, в UDE сгенерирован класс, которого не было в Docomatic, или наоборот) должны валидироваться через секцию `completeness_allowances` в JSON допущений.

### Шаг 3.6. Семантическое сравнение страниц контента (Semantic Content Match)
Для каждой сопоставленной страницы класса/интерфейса/метода:
1. **Экстракция текстовых блоков**:
   * Извлекаются семантические блоки страницы с помощью `loader.load_entity_content`.
2. **Нормализация текста**:
   * Для текстовых блоков Docomatic и UDE удаляются HTML-теги оформления (стили, шрифты, разметки колонок).
   * Удаляются дублирующиеся пробельные символы и пустые переводы строк.
   * Текст приводится к единому регистру и плоскому формату Markdown (CommonMark).
3. **Сравнение**:
   * Выполняется поабзацное текстовое сравнение.
   * Несовпадающие абзацы или отсутствующие описания параметров сверяются с секцией `content_allowances` файла допущений.
