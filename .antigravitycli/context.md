# Universal Document Engine (UDE) — Контекст Проекта

## 🎯 Суть проекта
Универсальное ПО для генерации документации. Принимает исходный код (через выгрузку Doxygen XML), парсит его структуру и комментарии, преобразует в промежуточное представление (IR) и генерирует Markdown для SSG, HTML-страницы или структурированный JSON/XML.

## 🛠️ Технологический стек
- **Язык разработки**: Python
- **Стиль кодирования**: PEP 8 (стандарты из `.antigravitycli/styles/code_style.md`)
- **Ключевые библиотеки**: `lxml` / `xml.etree` (XML-парсер), `Jinja2` (шаблонизатор)
- **Ввод**: Doxygen XML
- **Тестирование**: pytest (стандарты из `testing_style.md`)

## 📐 Архитектура репозиториев и пайплайнов
Проект организован как «зонтичный» репозиторий со связанными через **Git Submodules** подмодулями:

```text
Pipeline/ (Зонтичный репозиторий)
├── design-docs/            # [Git Submodule] https://github.com/Sir-Derryk/ude-design-docs.git
│                           # Пайплайн №1: Проектная документация (BRD, SRS, SDD) -> Docusaurus (с версионированием)
├── engine/                 # [Git Submodule] https://github.com/Sir-Derryk/universal-document-engine.git
│                           # Код Universal Document Engine (Python)
└── user-docs/              # [Git Submodule] https://github.com/Sir-Derryk/ude-user-docs.git
                            # Эксплуатационная документация и Портал (VitePress — премиальный дизайн)
```

### Пайплайны документации и триггеры:
1. **Пайплайн №1 (Проектный)**: 
   - *Триггер*: коммиты в подмодуль `design-docs/**` (или изменения в `.antigravitycli/**`).
   - *Результат*: Сборка проектной базы знаний в **Docusaurus** (изолированный поиск по спецификациям, поддержка версионирования архитектуры).
2. **Пайплайн №2 (Эксплуатационный)**: 
   - *Триггер*: коммиты в подмодуль `user-docs/content/**` (User/Admin Guides).
   - *Результат*: Сборка статических руководств в **VitePress** (премиальный дизайн, мгновенная SPA-навигация).
3. **Пайплайн №3 (API Reference)**: 
   - *Триггер*: коммиты в исходный код ПО (`engine/**`).
   - *Результат*: Сборка Doxygen XML -> запуск UDE -> генерация временных Markdown-файлов -> рендеринг в **Hugo** -> очистка временных файлов (высочайшая скорость сборки, изолированный поиск по классам/сигнатурам).

## ❓ Вопросы для проектирования BRD (Для обсуждения на следующем шаге)
1. **Главные «боли» (Pain Points) с существующими генераторами**: Почему стандартный вывод Doxygen (HTML), Sphinx или MkDocs-autodoc не устраивают? (Например: устаревший дизайн Doxygen, сложность кастомизации тем под современные SSG вроде Hugo, непереносимость документации между проектами).
2. **Главная бизнес-цель проекта**: Какую ключевую проблему процесса разработки или дистрибуции документации должен решить наш движок? (Например: автоматизация бесшовной доставки актуальных справочников API в изолированный портал документации без ручного труда).
3. **Предпочтительный целевой формат SSG**: Будем ли мы изначально строго ориентироваться на шаблоны для **Hugo** (учитывая, что это быстрый и популярный SSG), или сразу закладываем гибкие абстракции под MkDocs / Docusaurus?
