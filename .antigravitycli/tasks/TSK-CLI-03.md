# Задача: TSK-CLI-03 — Оркестратор пайплайна (Orchestrator)

## 📌 Часть 1: Инструкция по выполнению (Implementation Guide)
1. **Цель**: Создать центральный управляющий класс `UdeOrchestrator`, координирующий последовательное выполнение фаз сборки, гарантирующий стабильность пайплайна, полную портативность и относительное разрешение путей (соответствует Рекомендации 3 по развитию проекта).
2. **Шаги реализации**:
   * Создать файл `ude/orchestrator.py` с классом `UdeOrchestrator`.
   * **Требование к портативности путей (Relative Path Resolution / REQ-FUN-29)**:
     * Оркестратор обязан резолвить и переводить все относительные пути, указанные в конфигурационных файлах (`ude_config.json`, `ude_global.json`), в абсолютные пути **относительно расположения соответствующего файла конфигурации**, а не относительно текущей рабочей директории процесса.
     * Абсолютно запрещено хардкодить любые физические пути (например, `D:\My repositories\...` или конкретные папки диска C) в Python-скриптах ядра UDE.
   * Метод `run_pipeline(config_path: Path, format_override: Optional[str] = None)` должен:
     1. Загрузить конфигурацию целевого проекта (`ude_config.json`).
     2. Инициализировать коллектор (`DoxygenXmlCollector`), парсер (`DoxygenXmlParser`) и выбранный рендерер (`HugoMarkdownRenderer` или `HtmlRenderer`), передав им корректно вычисленные переносимые пути.
     3. Выполнить предохранительную валидацию окружения (`validate_environment()`).
     4. Запустить сбор ресурсов (`collect()`).
     5. Внутри блока `try...finally` запустить парсинг файлов в промежуточное представление IR (с проверкой кэша).
     6. В блоке `finally` обязательно вызвать очистку ресурсов коллектора (`cleanup()`), гарантируя удаление временной XML-папки даже при критическом сбое парсера.
     7. Передать полученный каталог IR на вход рендереру (с использованием кэша рендеринга).
     8. Поддерживать `error_policy` из глобальных настроек: прерывать работу сразу (`fail-fast`) или продолжать логирование предупреждений (`continue-on-error`).

## 🧪 Часть 2: Инструкция по проверке результата (Verification & TDD Scenarios)
1. **Тестовый сценарий (TDD Red Phase)**:
   * Написать `tests/test_orchestrator.py` с проверкой полного цикла пайплайна, обработки исключений и портативности путей.
   * Тест должен проверять, что запуск из различных рабочих директорий (`cwd`) при одинаковом пути к конфигурации дает идентичные корректные абсолютные пути к исходникам и результатам рендеринга.
   * Тесты должны упасть.
2. **Реализация (TDD Green Phase)**:
   * Написать класс `UdeOrchestrator`.
3. **Запуск и валидация (TDD Refactor Phase)**:
   * Запустить проверку:
     ```bash
     poetry run pytest tests/test_orchestrator.py
     ```
   * **Ожидаемый успешный результат**: тесты зеленые, оркестратор безупречно собирает и очищает временные папки, правильно резолвит относительные пути от места нахождения конфига и рендерит документацию.
