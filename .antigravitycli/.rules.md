# Local Workspace Rules

## 1. Clarification & Questions
- If there is any ambiguity, underspecified requirements, or if you have any technical/design questions, **always ask the user first and wait for their response before executing the prompt** (unless explicitly instructed to proceed without asking).

## 2. File Creation, Modification & Consent Flow (Consent-First)
- **Consent-First Writing**: Under no circumstances should the assistant write, modify, or delete any files on disk (including source code, tests, and tracking logs like `stage_diary.md` or `milestone_reports.md`) without receiving the user's explicit prior consent in the chat.
- **Proposal Phase**: The assistant must first describe the intended file changes, code blocks, or log entries in the chat.
- **Execution Phase**: Only after the user gives approval (e.g., in Russian: "Да", "Создавай", "Применяй" or in English: "Yes", "Create", "Apply"), the assistant is allowed to write the changes to disk. If the user refuses, the files remain untouched.
- **Temporary Files Cleanup**: For any temporary, scratch, or intermediate files created during development that are no longer needed, the assistant must explicitly list them, ask the user for permission to delete, and only delete them upon confirmation.

## 3. Artifact Storage
- By default, all requested artifacts must be stored in the project's root folder unless specified otherwise.

## 4. Git Workflow Automation & Approvals
- **Strict Command Constraint**: Under no circumstances execute `git commit` or `git push` without a direct, explicit command from the user.
- **Git Branch Protection & Submodule Policy**:
  - **Submodules (`engine`, `user-docs`, `design-docs`)**: Protected on the remote (`main` branch is protected). All modifications MUST be done via local task branches, pushed to remote, and merged only via GitHub Pull Requests after status checks pass. Direct pushes/commits to remote `main` are forbidden.
  - **Parent Repository (`Pipeline`)**: The main branch (`main`) accepts direct commits and pushes from the assistant/user (e.g. for submodules pointer updates, local configs, or metadata synchronization) without requiring a Pull Request, following explicit user consent.
  - **Submodule Pointer Update Order**: When modifying submodules, the precise chronological order of operations must be:
    1. Create a local task branch inside the submodule.
    2. Commit and push modifications to the submodule's remote repository.
    3. Create and merge a Pull Request into the submodule's protected branch (`main`/`master`) on the remote hosting platform.
    4. In the parent repository, check out the target branch and run `git submodule update --remote` (or pull the merged commit).
    5. Commit and push the updated submodule pointer inside the parent repository (`Pipeline`), following explicit user consent.
- **Sequence of Actions**:
  1. Assistant proposes file changes and log entries in the chat.
  2. User approves or denies.
  3. If approved, assistant writes files to disk.
  4. Assistant runs `git add`, `git commit`, and/or `git push` as requested by the user.
- **Branch Cleanup**: After successfully merging a Pull Request (PR) on the remote hosting platform, the assistant/user must delete the redundant or unneeded local and remote task branches to maintain repository cleanliness.
- **Service Files Verification**: The assistant must verify whether both `.antigravitycli/context.md` and `.antigravitycli/active_plan.md` require updates to reflect the physical directory structure, current milestone, and any changes on disk at least at the end of each development stage (before creating a git push/submitting milestone reports).

## 5. Code Development & Quality Standards
- **Style Adherence**: During code design and implementation, the assistant must strictly follow all development standards, code styles, architectural paradigms, and testing guidelines documented under the [`styles/`](file:///D:/My repositories/Pipeline/.antigravitycli/styles) directory (e.g., Python PEP 8 with Black/Flake8, TypeScript ESLint/Prettier with camelCase, OOP, TDD).
- **Environment Isolation**: The Python virtual environment (Poetry or `.venv`) must be initialized, maintained, and executed strictly inside the `engine/` submodule root directory. Under no circumstances should dependencies be mixed or executed from the parent repository level, ensuring clean testing and build runtimes.

## 6. Custom Local Skills
- **Custom Skills Location**: All custom, reusable project-level skills are stored in the `.antigravitycli/skills/` directory.
- **Requirements Audit**: Whenever the user asks to perform an audit or verify requirements, the assistant must consult and strictly follow the SOP defined in [requirements_audit.md](file:///D:/My%20repositories/Pipeline/.antigravitycli/skills/requirements_audit.md).

## 7. Security & Confidentiality (Confidentiality Gating)
- **Authorized for Publication**: Only information, commits, pushes, and documentation directly related to **Mock SDK** and **UDE** are authorized to be published, committed, or pushed to any public/unprotected remote hosting services (e.g., GitHub).
- **Strict Confidentiality of Real SDKs**: Under no circumstances should any information, code, metadata, or documentation regarding other projects (such as real proprietary SDKs: `facetmodeler`, `bimnv`, etc.) be pushed or committed to any unprotected remote. These projects must remain strictly local until a secured and protected environment (e.g., self-hosted GitLab) is established.
